    # MVC和MVVM

    前端架构模式，通过分离关注点来改进代码组织方式，一种架构模式往往能使用多种设计模式.

    # MVC

    全名是 Model View Controller，代表模块-视图-控制器的缩写。是一种将业务逻辑、数据、界面相分离的组织结构。

    Model: 是用于封装和应用程序业务逻辑相关的数据以及处理数据的方法。

    View: 作为视图层，主要是要来在页面展示数据, 通常依据模型数据创建.

    Controller: 作为模型和视图之间的纽带存在，Controller 把 Model 数据赋予 View 来显示，或者将 View 接收到的用户输入把这些数据传给 Model 来保存到本地或服务器。MVC 将响应机制封装在 Controller 之上。

    交互：

    1. view和Controller 就比如页面上点击事件，由 view 来接收的，但是处理这个事件的是Controller，View 只负责传递事件，但不关系命中谁，这是 View 和 Controller 的一种交互方式，叫 action-targer。

    2. Model和Controller Model 可以看作是数据的管理者，一旦数据发生了变化，View 的显示数据就应该发生变化，它们之间是一种同步关系，Controller 就会发生作用。

    优点：耦合性低、重用性高、生命周期成本低、可维护高。

    缺点：

    1. MVC 模式的业务逻辑主要集中在 Controller，前端的 View 其实已经具备了独立处理用户事件的能力，当每个事件都流经Controller时，这层会变得十分臃肿，视图与控制器相互分离却紧密连接，妨碍了它们的独立性。

    2. Model负责数据的取和存，但是除了存和取之外，还有一个重要的操作：解析。比如：可能需要把原始数据转为View才能使用的数据，就需要进一步的解析，把不属于 controller 的操作放到 控件里。为了更近一步的抽分，也就迫使调整架构。

    3. 视图对模型数据的低效访问。

    # MVVM

    除了上述架构确实是缺点的，其实随着前端项目的越来越复杂，混杂 html、css、js ，代码的组织和维护难度更加复杂。视图结构也需要进行进行调整。

    早期我们将数据写入节点中，我们需要操作 DOM 节点，使用浏览器提供的原声 API。

    随着 jQuery 的横空出世，我们可以使用简洁的 API 来操作 DOM，更加的轻巧。但是也需要我们进行操作 DOM。

    MVVM：

    1. Model: 可以称为是数据层，仅仅关注数据本身。
    2. view: 通过使用模版语法来声明式的将数据渲染进 DOM。当ViewModel对Model进行更新的时候，会通过数据绑定更新到View
    3. ViewModel: 业务逻辑也主要集中在这里, ViewModel负责把Model的数据同步到View显示出来，还负责把View的修改同步回Model。

    这样我们就可以不用关注DOM结构，把关注点分在代码逻辑上即可，关注点分离。

    当用户操作了界面，如果需要进行业务的处理，都会进行网络请求，去请求后端的服务器，此时，我们的1这个请求，就会被后端的 APP.js 监听到。Controller 不是不存在了，只是弱化了它的功能，Controller 来持有 VM。这样可以使得架构功能更加清晰。

    优点：低耦合、可重用性、独立开发。
